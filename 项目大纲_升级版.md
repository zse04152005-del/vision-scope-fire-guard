# 基于改进 YOLOv8 的校园火警多路智能监控系统（升级版项目大纲）

> 目标：在资源有限条件下先落地“中心单机 + 预留分布式”方案，实现 12 路视频大屏监控与 1–2 秒内告警响应；后续可平滑升级到多边缘节点 + 中央大屏。

## 1. 项目定位与范围
### 1.1 项目定位
- 场景：高校校园（实验室、宿舍楼、走廊、仓库等）
- 形态：多路视频大屏 + 统一告警中心
- 目标：低成本、低延迟、高可用、可扩展

### 1.2 In-Scope
- 多路视频接入（12 路为默认规模）
- 实时烟火检测与告警联动
- 大屏监控与告警中心
- 事件日志与截图归档
- 本地/远程通知（钉钉/企业微信/邮箱之一）
- 性能优化与可靠性增强

### 1.3 Out-of-Scope（当前阶段）
- 与消防局政务系统正式对接
- 嵌入式硬件部署（如 Jetson、树莓派）
- 商业级 7x24 运维

## 2. 总体架构（中心单机 + 预留分布式）
### 2.1 当前落地形态（Phase-1）
- 中心主机：Windows + RTX 4060（作为推理与大屏主控）
- 输入：12 路 RTSP/IP 摄像头（USB 仅用于临时/实验）
- 输出：本地大屏 + 告警通知 + 事件归档

### 2.2 未来演进形态（Phase-2）
- 边缘节点：每栋楼/区域一台普通电脑采集 + 推理
- 中央大屏：仅接收“事件 + 低码率预览流”并统一展示
- 架构接口：保留事件上报 API 与统一告警聚合层

## 3. 功能模块与数据流
### 3.1 模块拆分
1) 采集层（Capture Service）
- RTSP/USB 视频接入、断线重连、最新帧缓存

2) 推理层（Inference Service）
- YOLOv8n + CBAM + WIoUv3
- 多线程/队列：每路取最新帧，防止积压
- 动态降帧 + 分辨率自适应

3) 告警层（Alarm Service）
- 规则：置信度阈值 + 连续 N 帧 + 滑动窗口
- 去抖与重复抑制（同一摄像头短时不重复报警）
- 等级判定：疑似 / 预警 / 确认

4) 展示层（Display Service）
- 12 路大屏网格
- 统一告警中心：时间、摄像头、等级、截图
- 历史告警检索与回放入口

### 3.2 数据流
摄像头 → 采集 → 推理 → 告警判定 → 事件落库/截图 → 大屏与通知

## 4. 大屏与前端体验设计（核心优化）
### 4.1 自适应大屏布局（默认 12 路）
- 布局比例：顶部状态栏 10% + 主体区域 90%
- 主体区域：左侧 3x4 视频网格（约 75% 宽）+ 右侧告警中心（约 25% 宽）
- 自适应规则：根据屏幕尺寸自动缩放；摄像头少于 12 路自动居中填充
- 画面比例：保持 16:9 或接近原始比例，避免拉伸

### 4.2 线框草图（概念）
```
┌──────────────────────── 顶部状态栏 ────────────────────────┐
│ 在线路数 | 今日告警 | GPU/CPU | 网络状态 | 时间            │
├──────────────────────── 主体区域 ──────────────────────────┤
│ [Cam01][Cam02][Cam03][Cam04]  │  告警中心(列表+时间线)     │
│ [Cam05][Cam06][Cam07][Cam08]  │  - 时间 / 摄像头 / 等级    │
│ [Cam09][Cam10][Cam11][Cam12]  │  - 预览截图 / 操作按钮     │
└──────────────────────────────────────────────────────────┘
```

### 4.3 组件清单（前端 UI 结构）
- 顶部状态栏：在线路数、今日告警、GPU/CPU 占用、网络状态、系统时间
- 视频卡片：画面、摄像头名称、状态灯、实时 FPS、当前目标数
- 告警中心：告警列表、告警等级标签、截图预览、确认/忽略按钮
- 控制区（隐藏或弹出）：全部启动/暂停、单路重连、告警阈值设置

### 4.4 交互流程设计
- 一键启动/暂停：全部摄像头统一开关，显示连接状态变化
- 单路放大：双击视频卡片进入“单路聚焦”，再次双击返回网格
- 告警处置：点击告警 → 弹出详情卡片（时间、位置、置信度、截图）
- 误报标注：告警详情中“误报”按钮 → 写入日志，便于模型迭代

## 5. 告警闭环设计（核心优化）
### 5.1 告警分级策略
- 低置信度 + 短持续：疑似
- 中置信度 + 连续 N 帧：预警
- 高置信度 + 连续 M 帧：确认

### 5.2 通知策略
- 本地：大屏红色高亮 + 声音警报 + 弹窗
- 远程：钉钉/企微机器人，发送文本 + 截图
- 频率控制：同一摄像头 N 分钟内只推送一次确认级告警

### 5.3 事件留痕
- 自动截图：原图 + 标注图
- 告警日志：时间、摄像头、置信度、等级、处理状态
- 可导出报表（CSV/Excel）

## 6. 性能与稳定性
- 目标响应：1–2 秒内出现告警提示
- 每路推理帧率：5–10 FPS（足够快速响应）
- 动态策略：资源紧张时降帧或降分辨率
- 断线重连：RTSP 超时自动重连
- 容错：任一路掉线不影响整体系统

## 7. 部署与环境
- Windows 4060 作为中心主机
- Python + PyTorch + Ultralytics + PyQt6
- 模型文件统一放置到 models/ 目录
- 配置文件（摄像头地址、阈值）采用 JSON/YAML

## 8. 测试与验收
### 8.1 功能测试
- 多路接入与切换
- 告警触发/去抖
- 日志与截图保存

### 8.2 性能测试
- 12 路并发推理帧率
- 告警响应时间（<=2s）

### 8.3 误报/漏报测试
- 干扰场景（光照变化、烟雾机、视频噪声）

## 9. 里程碑计划（建议）
1) M1（1 周）：多路接入 + 大屏 UI 基础
2) M2（1–2 周）：告警中心 + 事件日志闭环
3) M3（1 周）：性能优化 + 稳定性与断线重连
4) M4（1 周）：测试报告 + 演示视频

## 10. 风险与应对
- 误报过多：引入负样本 + 连续帧判定 + 可调阈值
- 性能不足：降帧 + 降分辨率 + GPU 优化
- RTSP 不稳定：自动重连 + 心跳检测
